# 運用ルール索引と実行ゲート

## 1. 目的

本書は、portfolioリポジトリの横断運用規程を ChatGPT が作業開始時に参照しやすくするための入口文書である。

portfolioは公開ポートフォリオ用リポジトリであり、公開文言、職務経歴表現、制作物説明、公開安全性方針を扱う。そのため、一般的な実装完了条件に加えて、公開安全性条件を維持することを最優先にする。

本書は詳細規程を置き換えない。判断に迷う場合は、本書を入口として詳細規程を確認し、より厳格な規程を優先する。

---

## 2. 最優先原則

以下は全作業に優先する。

1. 要求仕様、ユーザー指定文言、公開文言、制作物説明、テスト期待値、運用規則を弱めない。
2. 安全チェックやツール制約を通す目的で、仕様を曖昧化、マイルド化、部分削除しない。
3. 住所、電話番号、生年月日、年齢、履歴書原本、スキルシート原本を配置しない。
4. 実企業名、顧客名、内部システム名、守秘義務に触れる詳細仕様を配置しない。
5. 一部反映、未コミット、未push、PR未作成、CI未確認、main未反映を完了扱いしない。
6. 実装済み・未実装・一部反映・未反映・未確認を明確に区分して報告する。
7. Codex、ChatExec方式、ユーザー手作業へ切り替える場合も、要求仕様と公開安全性条件を維持する。
8. `main` への直接push、秘密情報の出力・コミットを行わない。

---

## 3. 実行主体分類

| 実行主体 | 担当範囲 | 主な判断基準 |
|---|---|---|
| ユーザー(人) | 最終判断、公開判断、外部サービス操作、merge判断、主観評価 | AIが代替できない判断または公開可否判断が必要な場合 |
| ChatGPT(会話内) | 整理、レビュー、比較、計画、Codex指示書作成、作業分解 | 会話内で完結し、リポジトリ状態を直接変更しない場合 |
| ChatGPT(リポジトリ編集) | 小〜中規模文書修正、README更新、docs整理、軽微なHTML/CSS修正、PR確認 | 対象ファイルを完全取得でき、差分が局所的で安全な場合 |
| Codex | 複数ファイル修正、HTML/CSS/データ整合、テストまたは表示確認、PR作成 | 横断修正や検証が必要な場合 |

---

## 4. 作業開始ゲート

作業開始時、ChatGPTは次を確認する。

1. Memory上のプロジェクト運用ルール。
2. 本書 `00_運用ルール索引と実行ゲート.md`。
3. 対象タスクに関係する詳細規程。
4. Open PRの有無。
5. 直前タスクのmerge、課題管理整理、リモート作業ブランチ削除、ローカルprune前提。
6. 作業主体分類。
7. 公開安全性条件が維持されていること。
8. 未実装機能や未確認事項を実装済み・確認済みとして扱わないこと。

---

## 5. Codex投入ゲート

Codexへ作業を渡す前に、ChatGPTは以下を確認する。

| 確認項目 | 必須内容 |
|---|---|
| モデル設定 | モデル、インテリジェンス / reasoning effort、速度を明記する |
| ブランチ方針 | 最新 `main` から新規クリーンブランチを作る。旧ブランチ継続利用可否を明記する |
| 対象ファイル | 主対象、確認対象、変更禁止対象を分ける |
| 要求仕様 | 正確に追加、削除、変更すべき仕様を列挙する |
| 公開安全性 | 個人情報、実企業名、顧客名、内部仕様、守秘情報を含めない確認観点を明記する |
| 安全制約 | 仕様、文言、公開安全性条件を弱めない必須文言を含める |
| 確認手順 | 表示確認、リンク確認、ビルド、CIなどを明記する |
| Git公開Step | `git add`、`git commit`、`git push`、PR作成を明記する |
| 完了条件 | commit、remote push、PR作成、確認結果、未反映事項を必須にする |
| 結果報告 | 統一ステータス、PR番号、commit SHA、CI状態、未反映事項を含める |

Codexへmerge、課題管理整理、merge後ブランチ削除を混ぜてはならない。

---

## 6. PR未作成時の追補優先ルール

Codexが仕様反映と確認手順を完了しても、commit、remote push、PR作成が未完了の場合、その作業は完了扱いしない。

ChatGPTは、ユーザー手作業へ直行せず、原則として以下の順に対応する。

1. Codexへcommit、push、PR作成の追補指示を出す。
2. Codexが安全チェック、権限、remote、CLI、認証、GitHub連携制約で失敗した場合、ChatExec方式またはChatExec2方式へ切り替える。
3. ChatExec方式でも不適切または失敗した場合、ユーザー手作業へ切り替える。
4. いずれの場合も要求仕様、ユーザー指定文言、公開安全性条件を弱めない。

同一目的でCodex追補が2回失敗した場合は、ChatExec方式またはユーザー手作業へ切り替える。

---

## 7. 統一完了状態定義

| 状態 | 定義 | 完了扱い |
|---|---|---:|
| `LOCAL_SUCCESS` | ローカル差分作成とローカル確認PASS。ただし未commit、未push、またはPR未作成 | いいえ |
| `COMMITTED` | commit済み。ただしremote未push | いいえ |
| `PUSHED` | remote branchあり。ただしPR未作成 | いいえ |
| `PR_READY` | PR作成済み。CIまたは表示確認未確認 | いいえ |
| `CI_GREEN` | CIまたは必要確認がsuccess。レビュー・merge判断待ち | いいえ |
| `DONE` | main merge済み。必要な課題管理整理、ブランチ整理、prune確認まで完了 | はい |
| `PARTIAL` | 一部反映、未反映事項あり | いいえ |
| `BLOCKED` | 安全チェック、権限、CLI、認証、CI等で停止 | いいえ |

Codexや外部AIが `STATUS: SUCCESS` と報告しても、PR未作成、CI未確認、main未反映がある場合、ChatGPT側では上表に従って再分類する。

---

## 8. ChatExec / ChatExec2切替ゲート

ChatExec方式またはChatExec2方式は、以下の場合に使用する。

- ChatGPT(リポジトリ編集)の直接反映が安全チェックまたはツール制約で不安定な場合。
- Codexがcommit、push、PR作成で停止し、同一目的の追補でも解決しない場合。
- ユーザーPC上のローカル差分、表示確認、CLI実行結果を安全に確認する必要がある場合。
- 並行作業やPR/Issueコメント方式が有効な場合はChatExec2方式を優先する。

---

## 9. PR / CI / mergeゲート

PR作成後、ChatGPTは以下を確認する。

1. PR base が `main` であること。
2. PR head が作業ブランチであること。
3. 変更ファイルがスコープ内であること。
4. 要求仕様が弱められていないこと。
5. 公開安全性条件が維持されていること。
6. 必要な表示確認、リンク確認、CIが success であること。
7. 未反映事項がない、または明示されていること。
8. merge判断がユーザー(人)の責務として残っていること。

---

## 10. Codex結果報告フォーマット

```text
JOB_ID:
STATUS: LOCAL_SUCCESS | COMMITTED | PUSHED | PR_READY | CI_GREEN | DONE | FAIL | PARTIAL | BLOCKED
Repository:
Branch:
Commit:
Remote Push: DONE | NOT_DONE | BLOCKED
PR: #number | NOT_CREATED | BLOCKED
Local Check:
CI:
Changed Files:
Public Safety Check:
Requirement Mapping:
Unreflected Items:
Blocking Point:
Next Required Action:
```

`PR: NOT_CREATED` または `Remote Push: NOT_DONE` の場合、`STATUS: DONE` または `STATUS: SUCCESS` として扱わない。

## 11. 詳細規程への参照導線

| 用途 | 参照規程 |
|---|---|
| 安全チェック・ツール制約・仕様不変切替 | `安全チェック発生時の仕様不変切替方針.md` |
| Codex指示書の標準形 | `Codex指示書テンプレート.md` |

本書と詳細規程に重複がある場合は、より厳しい完了条件、より明確な禁止事項、より仕様不変性と公開安全性を強く保つ規程を優先する。
