# 安全チェック発生時の仕様不変切替方針

## 1. 目的

本方針は、portfolioリポジトリにおけるChatGPT(リポジトリ編集)、Codex、ChatExec方式、ユーザー手作業の運用時に、安全チェックまたはツール制約が発生した場合の判断基準を定める軽量版の運用規程である。

portfolioは公開ポートフォリオ用リポジトリであり、公開文言、職務経歴表現、制作物説明、公開安全性方針を扱う。そのため、安全チェックを通すことを目的に、ユーザーが指定した表現、掲載方針、公開安全性条件、タスク管理方針を勝手に弱めたり、曖昧化したり、別表現へ置き換えたりしてはならない。

## 2. 基本原則

安全チェックまたはツール制約が発生した場合、次の原則を守る。

1. 要求仕様、ユーザー指定文言、公開文言、制作物説明、テスト期待値、運用規則を安全チェック通過目的で弱めない。
2. 仕様を曲げた代替表現で通そうとしない。
3. 反映できた範囲と反映できなかった範囲を明確に分けて報告する。
4. 同一目的の直接編集が連続してブロックされた場合、ChatGPT(リポジトリ編集)での直接反映を打ち切る。
5. Codex、ChatExec方式、またはユーザー手作業など、仕様を保ったまま反映できる別経路へ切り替える。

## 3. portfolio固有の注意点

portfolioでは、公開安全性を最優先する。

- 住所、電話番号、生年月日、年齢、履歴書原本、スキルシート原本を配置しない。
- 実企業名、顧客名、内部システム名、守秘義務に触れる詳細仕様を配置しない。
- 公開制作物の説明を、安全チェック通過目的で過度に曖昧化しない。
- 職務経歴や実績の表現を、ユーザー確認なしに実態と異なる表現へ置き換えない。
- 公開安全性のために削除または抽象化が必要な場合は、削除理由と影響範囲を明示する。

## 4. 禁止事項

以下を禁止する。

- 安全チェックを通すために、要求された表現を勝手にマイルド化すること。
- 仕様上必要な記述を削除すること。
- 公開文言の意味が変わる代替表現で通そうとすること。
- 一部だけ反映された状態を完了済みとして報告すること。
- 直接編集が不安定な状態で、同じ操作を表現だけ変えて反復すること。
- ブロック回避を優先して、ユーザーが求めた仕様と異なる成果物を作ること。

## 5. 状態報告の必須区分

安全チェックまたはツール制約が発生した場合、以下の区分で状態を報告する。

| 区分 | 報告表現 |
|---|---|
| コミット済み | 反映済み |
| ツールでブロック | 未反映 |
| 一部のみ反映 | 一部反映、未完了 |
| テスト未更新 | CI未確認、merge不可 |
| PR未作成 | main反映不可 |
| 仕様どおりに反映できない | 仕様不変の別経路へ切替 |

## 6. 切替基準

以下のいずれかに該当した場合、ChatGPT(リポジトリ編集)での直接反映を中止し、別経路へ切り替える。

1. 同一目的の `update_file`、`create_file`、`create_pull_request`、`update_pull_request` などが2回以上ブロックされた場合。
2. ブロック回避のために表現変更が必要になり、仕様の意味が変わる可能性がある場合。
3. HTML、CSS、データ、docs、TSK管理など複数ファイルの整合確認が必要で、直接編集では検証が不十分な場合。
4. ブランチ状態がdiverge、競合、既mergeブランチ再利用などで複雑化した場合。
5. ユーザー指定文言を正確に保持する必要がある場合。

## 7. 切替先の優先順位

| 優先度 | 切替先 | 用途 |
|---:|---|---|
| 1 | Codex | 複数ファイル修正、HTML/CSS/データ整合、テストまたは表示確認が必要な場合 |
| 2 | ChatExec方式 | ローカルで差分確認、表示確認、簡易チェックを行う場合 |
| 3 | ユーザー手作業 | 指定文言を正確に反映する必要があり、AI側ツールが不安定な場合 |
| 4 | GitHub Issue/PRコメントによる指示化 | 直接反映せず、仕様と未反映事項を管理する場合 |

## 8. Codexへ切り替える場合の必須指示

Codexへ切り替える場合、指示書には次を必ず含める。

> 安全チェックやツール制約を回避する目的で、要求仕様、文言、テスト期待値、公開安全性条件を弱めたり曖昧化したりしないこと。指定どおりに反映できない場合は、代替表現で済ませず、未反映として報告すること。

あわせて、以下を明記する。

- 正確に反映すべき仕様。
- 変更対象ファイル。
- 変更してはならない範囲。
- 公開安全性の確認観点。
- 実行する確認手順。
- 完了報告に含めるべき差分、確認結果、未反映事項。

## 9. 完了判定

安全チェック発生後に別経路へ切り替えた作業は、次の全てを満たすまで完了扱いしない。

1. 要求仕様が弱められずに反映されている。
2. 対象ファイルの差分が確認されている。
3. 公開安全性条件が維持されている。
4. 必要な表示確認、リンク確認、またはCIが成功している。
5. PRが作成され、必要に応じてmerge可能状態である。
6. 未反映事項がない、または未反映事項が明示されている。
